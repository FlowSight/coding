Review Parser
Input: review="I booked a house on Airbnb for my trip to San Francisco. It was a lovely experience."
map={"Airbnb": "business", "san francisco": "city"}
Output:
"I booked a house on [business]{Airbnb} for my trip to [city]{San Francisco}. It was a lovely experience."
需要问的corner case有：
What if map.Keys are case sensitive？Like include (“Airbnb”, “business”) and (“airbnb”, “company”)
When a word can match multiple keys, should the longer key or the shorter key be used? For example, {"san francisco": "city", "san": "name", francisco": "name"}.
Should we replace the substring if it is the key? (It should not be replaced; only the complete word or phrase should be replaced.)
If the longest match is discarded because it is a substring, how should the shorter matches be read? Should all matches be stored or should boundary checks be performed beforehand?
Solution: Convert the map into a case-sensitive trie, prefer the longest match, and only replace whole words/phrases.
Follow-up: If the dataset is very large or there are long common prefixes among the words, the original character-based key can be changed to a prefix-based key, which can save space by reducing the number of trie nodes.
Booking Airbnb
When booking a trip on Airbnb, if we do not have a place available for all of the requested dates, we will suggest a Split Stay where you stay at two different Airbnbs, one for the first part of your trip and the other for the remainder. Given a list of Airbnbs with availability, make an api endpoint that will return all possible combinations of Split Stays across two listings for a given date range For this interview question, we will give each listing a name ("A", "B", etc) and our availability will just be a list of day numbers. For example: Airbnb A - [1,2,3,6,7,10,11] Airbnb B - [3,4,5,6,8,9,10,13] Airbnb C - [7,8,9,10,11] Given the date range [3-11], the expected result would be all sets of two Airbnbs that could form a split stay: Results: [ [B, C] Clarify the performance requirements. If better than O(n²) is required, the approach is to create two SortedLists, `moveOut` and `moveIn`, where the key is the date and the value is the ID. `moveOut` records the latest move-out date for Airbnb i. `moveIn` records the earliest move-in date for the end date. If the move-out date for Airbnb x is greater than or equal to the move-in date for Airbnb y, then x and y can be combined.












Design Retrier
You need to confirm the requirements with the interviewer, including which cases need to be covered, whether a retrier class or method needs to be implemented, whether it's a synchronous or asynchronous retrier, how the retrier will be called, and whether generic method support is required. Ask the interviewer to provide some examples, etc. Only after confirming the requirements should you start designing the retrier's arguments.
Case 0: Return with successful result;
Case 1
: Retry with max attempt and fixed delay; Case 2: Retry with fixed/increase delay and deadline; Case 3 :
Exponential back off retry; Case 4: Retry with certain exception; Case 5: Retry with certain result; Case 6: Return result/exception when retry is exhausted.  Example of calling: `String res1 = withRetry(() => "Success"); String res2 = withRetry(() => throw new RuntimeException("Always fail")); ` This example illustrates that the retrier is a generic method that includes optional arguments such as "maxAttempts" and "delay". It returns the result on success and the final exception when retry attempts are exhausted.







OOD1
Implement a class that supports the following operations:
set_value(key: str, value: int): Stores the integer value for the given key.
set_sum(key: str, values: List[str]): Defines key as the sum of the values ​​associated with the keys in values.
get_value(key: str) -> int: Returns the integer value associated with key. If the key was defined using set_sum, return the computed sum.
There is a relationship between node and node. Please pay attention to update dependencies. The detailed examples are as follows:
set_value("A", 5): Sets A=5
set_value("B", 10): Sets B=10
set_sum("C", ["A", "B"]): C depends on A and B, so C=15
set_sum("D", ["C", "C", "A"]): D depends on C and A, so D=15+15+5=35
get_value("A"): Returns 5
set_value("B", 10): B already 10, No change needed.
`get_value("C")`: Returns 15.
`get_value("D")`: Returns 35.
`set_value("A", 100): A changes to 100, triggers updates:
C updates to 100+10=110.
D updates to 110+110+100=320.
`get_value("D")`: Returns 320.
Clarify whether there are duplicates in the `Children` class. If so, only a hash table can be used; otherwise, a set can be used. Also, determine if input validation is needed, etc.
Follow-up question: If `get_value` is not frequently used and `set_value` is frequently used, how to improve the time complexity of `set_value`? 
Answer: Put the `calculate sum` function inside the `get` function, counting each call as one, or add an `expire` flag to recalculate after modifications.
OOD isn't difficult; the main thing is to consider the design's rationality. Before writing, write down the logic of all cases and communicate clearly, then implement the logic block by block when coding.

I unexpectedly got a code review interview during my online interview.
Beforehand, I thought it would be like debugging a LeetCode problem to find bugs. Instead, they gave me three code reviews/reviews, each involving changes to many files, just like a typical code review. My first review was about adding pagination, the second was about adding a check for valid values, and the third was about batch calling. Many requests came in, were put into a queue, and only when a batch was full were they called on an external API—modifying the queue and calling the external API concurrently. You had to find and comment on the code, such as missing null checks, variables that should be statically final but were created new ones in methods, races due to concurrency, etc. Another point is that this round didn't involve interaction with the interviewer, unlike the coding and SD rounds where you constantly exchange ideas. My interviewer was an Indian guy. He threw three code snippets at me, then turned off his camera and didn't say anything. I stood there stunned for a long time, wasting a lot of time. Plus, with three snippets to review, I wasn't sure how much time to allocate for each, so I became quite flustered. Looking back, the code was actually quite simple; I could have reviewed it like I do in my daily work. But at the time, my mind went blank, and I missed several obvious bugs. I only realized this later when I approached it with the mindset I normally use for reviews. It's a pity. The other rounds of coding, SD, and behavioral questions went very well with both the Chinese and white interviewers.
I'm sharing this with friends who are interviewing with them recently, so they won't be caught off guard by seemingly simple questions.

One involves adding pagination functionality to a system, another involves checking the validity of input data, and the third involves designing a batch processing call to an external API, which involves concurrent operations. Please conduct a comprehensive review of these codes and identify any potential issues, such as missing null checks, a variable that should be declared static final but is not, or potential thread safety issues in concurrent operations. Leave comments on the lines where you identify the issues. Note: This interview does not involve direct interaction with the interviewer.
